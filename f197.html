Day 1 - Wish List

  • in_array() 函数缺陷

漏洞点: 在第17 行中, in_array() 函数的第三个参数strict 未设置为TRUE 由于PHP 弱类型,导致了任意文件上传漏洞.

在 $haystack 中搜索 $needle ，如果第三个参数 $strict 的值为 TRUE ，则 in_array() 函数会进行强检查，检查 $needle 的类型是否和 $haystack 中的相同。如果找到 $haystack ，则返回 TRUE，否则返回 FALSE。

攻击者可以通过构造的文件名来绕过服务端的检测，例如文件名为 7shell.php 。因为PHP在使用 in_array() 函数判断时，会将 7shell.php 强制转换成数字7，而数字7在 range(1,24) 数组中，最终绕过 in_array() 函数判断，导致任意文件上传漏洞。（这里之所以会发生强制类型转换，是因为目标数组中的元素为数字类型）php > var_dump(‘7shell’ == 7);
bool(true)

第一个洞是看了分析文章才找到的,忘了代码审计的最好方法了,即去找危险函数和功能点…而不是在翻译程序.

CMS 实例分析: piwigo-2.7.1 (一个图库CMS)

真的是漏洞吗? -piwgo-2.7.1\admin\include\functions_upload_inc.php

在第 244 行中,使用了in_array()函数检查文件的后缀是否在合规范围内,但是因为没有使用第三个参数,但是别太早下论断, 因为前面的代码使用了getimagesize 来判断图片,图片类型,这里虽然可以通过图片文件头伪造来绕过 GIF89a 然后就会被判断为 GIF文件,后缀就被改为了 .gif , 故无法利用

因为这个函数涉及到判断 ,而判断这个一般就是判断用户输入,上传内容格式,大小,内容

所以就奔着这个功能去找就行了,上传不能利用,就找其他的地方

在include\functions_rate.inc.php 中,存在SQL注入漏洞

在方法 rate_picture 方法中, 42 行,判断$rate是否是存在于rete_items中的, 在piwigo\include\config_default.inc.php 中,即判断是否为,0,1,2,3,4,5 由于未设置第三个参数,导致了可以利用弱类型构造出' 来进入注入

, 跟进这个变量, 在111~122行的查询语句中,直接将$rate 拼接进入了(并且恰到好处的在最后一个位置),且$rate没有任何对数据进行过滤的措施.导致了SQL注入

将 $rate 设置为 1 and if(ascii(substr((select database()),1,1))=112,1,sleep(3))#即可进行盲注

如何修复?

因为这个漏洞是因为弱类型导致的,所以可以使用强匹配进行修复。例如将 in_array() 函数的第三个参数设置为 true ，或者使用 intval() 函数将变量强转成数字，又或者使用正则匹配来处理变量

Day 2 - Twig

第13 行使用了 Twig 模板引擎定义的 escape 过滤器来过滤link, 而实际上,这里的 escape 过滤器,是用的PHP 内置函数htmlspecialchars() 实现的

Internally, escape uses the PHP native htmlspecialchars function for the HTML escaping strategy.

第二处过滤在第23行,使用filter_var() 过滤中的FILTER_VALIDATE_URL 过滤器 ,来检查url是否为合法url

这里采用了 javascript伪协议来绕过 即 javascript://.... (利用了filter_var函数的缺陷)

这里对% 进行URL 编码成为 %25, // 表示单行注释符, %0a 表示换行,所以alert(1); 没有被注释掉

为什么要把% 编码成为 %25 呢, 因为浏览器现将payload解码为javascript://comment%0aalert(1) 再储存在变量url中

format 方法, format 方法, 对uri 进行了四次过滤,但是都未过滤XSS(未对标签进行转义)

payload:

http://127.0.0.1/anchor/index.php/<script>alert('xss!!!attack')</script>

当访问不存在的链接时, 就会调用 404.php 模块中的current_url()函数,获取当前的URL并将当前访问的页面(也就是index.php/以后的内容)输出到页面上,但是在过滤的时候并未处理XSS, 所以导致了反射型XSS 漏洞

  • XXE
  • 实例化任意对象漏洞

在第10 行的时候, 使用了class_exists() 函数来检查类是否定义,如果未定义,在存在__autoload() 函数的情况下, 函数默认会自动调用未定义的类, 在php 5.0 ~ 5.3 时,可以使用目录穿越 来实现任意文件包含

/?c=../../../../etc/passwd 即可实现读取/etc/passwd 是非常严重的代码设计不严格导致的漏洞

在第7~12 行的时候,我们可以看到,调用的类和参数都是可控的,虽然这段代码中没有可以利用的函数,但是我们可以利用PHP的原生类,来进行攻击. PHP 的 SimpleXMLElement 类,存在一个xxe 漏洞, 类似的可造成 xxe 漏洞的函数还有 DOMDocument() simplexml_load_string() 函数

https://github.com/Ifonly-go2019/vulhub/tree/master/php/php_xxe

环境条件:

php 7.0.30

libxml 2.8.0

复现: https://buuoj.cn/challenges#[PHP]XXE

实例: Shopware 5.3.3

在engine\Shopware\Controllers\Backend\ProductStream.php 中的 loadPreviewAction() 方法, 中,将传递过来的参数 sort 赋值给 sorting后 在 63 行对其进行反序列化, 其中 unserialize() 函数是通过 streamRepo repository类来调用的

repository类 中的 unserialize方法, 可以看到是通过调用 reflector 对象进行调用的,而 reflector 是通过LogawareReflectionHelper 类进行调用的

跟进LogawareReflectionHelper 类,找到 unserialize 方法

可以看到,这里将已经序列化后的值(这里的serialized 也就是 sort) 通过数组遍历对classname 和 arguments 进行一次赋值, 然后传入 createInstanceFromNamedArguments 方法, 跟进一下这个方法, 是在 ReflectionHelper 类中

在这个方法中,返回一个新的实例化对象,然而参数 可控,因此我们可以实例化一个SimpleXMLElement 类的对象,形成XXE漏洞

修复:

关于PHP中XXE漏洞的修复，我们可以过滤关键词，如： ENTITY 、 SYSTEM 等，另外，我们还可以通过禁止加载XML实体对象的方式，来防止XXE漏洞.关键修复代码 :

libxml_disable_entity_loader(true); 禁止加载外部实体

Day 4 - False Beard

  • strpos()函数缺陷

我们可以看到strpos 函数,如果在第一个参数中找到了我们要匹配的字符串,那么是返回该字符串出现的位置(int),如果不存在,返回false

在第 18,19 行就将用户的输入格式化为xml 进行存储, 在13,14 行,通过strpos() 来检查用户的输入中是否有 <,>,可能开发者也考虑了XML注入

开发者只考虑了为 false 的情况, 没有考虑出现在首字符为 0 的情况, 因为 false 和 0 取反 均为 true

payload: user=<"><injected-tag%20property="&pass=<injected-tag>

修复: 使用=== 强相等进行判断

Day 5 - Postcard

  • 绕过filter_var 函数对邮件的检测结合 mail()底层调用 escapeshellcmd()函数处理字符串
  • 结合escapeshellarg() 函数实现参数逃逸,导致远程代码执行

首先需要了解的是, 在Linux系统中,php 的内置函数mail 在底层中已经写好了,默认调用Linux的 sendmail程序发送邮件, sendmail 主要支持的选项有:

  • -O option = value

    QueueDirectory = queuedir 选择队列消息

  • -X logfile

    这个参数可以指定一个目录来记录发送邮件时的详细日志情况。

  • -f from email

    这个参数可以让我们指定我们发送邮件的邮箱地址。

在第7 行的时候,使用了filter_var()函数来检测邮件地址是否为合法地址

该函数存在一个问题, https://stackoverflow.com/questions/19220158/php-filter-validate-email-does-not-work-correctly

其中有个结论是 :

none of the special characters in this local part are allowed outside quotation marks ，表示所有的特殊符号必须放在双引号中。 filter_var() 问题在于，我们在双引号中嵌套转义空格仍然能够通过检测。同时由于底层正则表达式的原因，我们通过重叠单引号和双引号，欺骗 filter_val() 使其认为我们仍然在双引号中，这样我们就可以绕过检测

php 的mail() 函数在底层实现中,调用了escapeshellcmd()函数,对用户的输入的邮箱地址进行检测,即使存在特殊符号,也会被转义,

然后在第11行,返回值是经过escapeshellarg()函数处理的,这和之前的一道题中的漏洞点很相似

https://hack-for.fun/posts/20200223/#BUUCTF-2018-Online-Tool

因为这两个函数+ 正则表达式的原因,导致特殊字符逃逸就可以造成参数注入,致\\ ,前一个\用于转义,最后只有一个\

所以传入的参数: 127.0.0.1' -v -d a=1

  • escapeshellarg() 转义单引号

    参数变为: '127.0.0.1'\'' -v -d a=1'

  • escapeshellcmd() 转义特殊字符,转义\ 和 a=1’ 的单引号

    参数变为: '127.0.0.1'\\'' -v -d a=1\'

    最后的payload就是: curl 127.0.0.1\ -v -d a=1' ,即向127.0.0.1\ 发起POST请求,请求的数据为a=1’

修复:

传入的参数不接受 -f 参数, 如果不传递这个参数,payload 就不会被带入到mail函数, 不建议使用escapeshellarg 和 escapeshellcmd 来对参数进行过滤

利用可能性:

1、php version < 5.2.0 2、phpmailer < 5.2.18 3、php 没有安装 pcre（no default） 4、safe_mode = false（default）

存在正则绕过之后，以及 escapeshellarg 和 escapeshellcmd 一起使用造成的神奇现象之后。

只需要 phpmailer < 5.2.20

Day 8 - Candle

好文章: https://mochazz.github.io/2018/08/13/%E6%B7%B1%E5%85%A5%E7%A0%94%E7%A9%B6preg_replace%E4%B8%8E%E4%BB%A3%E7%A0%81%E6%89%A7%E8%A1%8C

  • preg_replace 在 /e 修饰符模式下的代码执行漏洞(PHP 5.5)
  • 漏洞原理: /e 修饰符模式下, 会将replacement字串中的PHP代码进行解析
  • 预备知识: 反向引用 和 可变变量

反向引用

payload: ?\S*={${phpinfo()}} => 正则表达式变成了/('.\S*.')/ei,'strtolower("\\1"),{${phpinfo()}}

为什么要匹配到 {${phpinfo()}} 或者 ${phpinfo()} ，才能执行 phpinfo 函数，这是一个小坑。这实际上是 PHP可变变量 的原因。在PHP中双引号包裹的字符串中可以解析变量，而单引号则不行。 ${phpinfo()} 中的 phpinfo() 会被当做变量先执行，执行后，即变成 ${1} (phpinfo()成功执行返回true)

修复方案

漏洞是 preg_replace() 存在 /e 模式修正符，如果正则匹配成功，会造成代码执行漏洞，因此为了避免这样的问题，我们避免使用 /e 模式修正符
